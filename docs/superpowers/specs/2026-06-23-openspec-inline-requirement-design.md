# gameai-openspec-from-doc 文字需求输入设计

## 目标

让 `gameai-openspec-from-doc` 除文档路径外，也能直接接收一段需求文字。文字完成输入归一化后，继续复用现有 Phase 2–4，不改变融合分析、OpenSpec artifacts 和 Explore 讨论流程。

## 调用方式

支持以下新增形式：

```text
/opsx:from-doc 做一个连续签到活动，连续 7 天，每天奖励递增
/opsx:from-doc -r "做一个连续签到活动，连续 7 天，每天奖励递增"
```

原有文档调用方式保持兼容：

```text
/opsx:from-doc 需求.docx
/opsx:from-doc -r 需求.docx -d 设计.docx -i 接口.docx
```

## 输入识别

Phase 1 先将需求输入归一化为 `$req_content`：

1. 输入对应一个存在的受支持文件时，按现有 `.docx`、`.md`、`.txt` 规则提取。
2. 输入不是有效文件路径但包含非空文字时，将完整文字作为 `$req_content`。
3. `-r` / `--requirement` 使用相同识别规则。
4. `-d` / `--design` 和 `-i` / `--interface` 仍只接受文档路径，本次不扩展为文字输入。
5. 没有有效需求文件或非空需求文字时，终止并提示用户补充需求。

文字输入不得因空格被拆成多个输入；应保留用户提供的完整内容。若内容看起来像文件路径但文件不存在，先提示路径不存在，再询问是改为按需求文字处理还是重新提供路径，避免把拼错的路径误当需求。

## 后续流程

- Phase 2：需求文字与文档提取文本均执行相同的 15 维度需求分析。
- Phase 3：从需求标题、首句或核心功能生成 kebab-case change name，并让用户确认。
- Phase 4：按现有方式展示方案摘要和待确认问题。
- 输出 artifact、融合规则、OpenSpec CLI 降级策略均保持不变。

## 输出标识

Phase 1 应明确显示输入来源：

```text
📝 需求文字：已接收 35 字
```

文档输入继续使用现有标识：

```text
📄 需求文档：签到-新版.docx — 75 段
```

融合分析中的来源统一标记为“需求输入”；需要细分时可标记为“需求文字”或“需求文档”。

## 异常与边界

- 纯空白文字视为未提供需求。
- 超短或语义不完整的文字不自行补全，进入待确认问题。
- 文件提取失败时，不自动把路径字符串当作需求正文。
- 同时提供需求文件和额外裸文字时，要求用户明确哪个是主需求，避免静默合并。
- 文字中的命令样式字符、换行和 Markdown 保持原意，不作为额外 CLI 参数解析。

## 验证

至少检查以下场景：

1. 裸需求文字能够进入 Phase 2。
2. `-r` 需求文字能够进入 Phase 2。
3. 原有单文件和多文件调用不受影响。
4. 不存在的疑似文件路径不会被静默当作需求。
5. 空文字会被拒绝。
6. 文字需求能够生成合理的 change name 候选。
